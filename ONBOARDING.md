# Onboarding: Emryk ML — for a Claude session working on the marketing site

You are reading this because you are about to work on the **public marketing website** for the Emryk Workstation product. This document gives you the context you need to write copy that is accurate, specific, and aligned with the product's actual posture — not generic security-theater.

The source of truth for everything in this document is the `rhuze-emryk/emryk-ml` GitHub repo. When in doubt, read the docs there directly:

- `README.md` — what the image is, what's in it, how it's used
- `SECURITY.md` — public-facing threat model + disclosure policy
- `KEY-POLICY.md` — signing-key lifecycle
- `SECURITY-TODO.md` — engineering hardening backlog (closed as of 2026-05-22)

## 1. What `emryk-ml` is

`emryk-ml` is a Fedora bootc image published to `ghcr.io/rhuze-emryk/emryk-ml`. It is the **foundation layer** of the commercial Emryk Workstation product — managed ML cloud workstations targeted at researchers, AI teams, and ML engineers who want a fully-provisioned NVIDIA + KDE + container workflow without doing the systems work themselves.

The image is open-source on GitHub but the *product* is the workstation service Emryk runs on top of it.

Two tags ship:

- `:latest` — base image, public foundation.
- `:latest-private-ml` — base + Mullvad VPN preconfigured. This is the customer-facing variant.

---

## 2. The trust story (this is the differentiator)

Every image published has **three independent supply-chain trust signals** attached. A customer's security team can verify any of them without trusting the other two:

1. **Cosign signature** (long-lived key, ours): proves we signed it. Required for `bootc` to even pull it on installed customer hosts (the image enforces this).
2. **SLSA build provenance** (Sigstore-signed, no long-lived secret): cryptographically proves *which GitHub workflow* built the image, from *which commit*, at *what time*. Verifiable with `gh attestation verify oci://ghcr.io/rhuze-emryk/emryk-ml:latest --repo rhuze-emryk/emryk-ml`.
3. **CycloneDX SBOM** (Sigstore-signed): complete machine-readable list of every package in the image, generated by syft from the published artifact (not from the source). Verifiable the same way with `--predicate-type https://cyclonedx.org/bom`.

**Why this matters for marketing copy:** "we sign our images" is generic. "Three independent attestations, any one verifiable by your security team in 30 seconds, no Emryk-side trust required" is specific and differentiating. Most ML cloud / GPU workstation products do *none* of this.

## 3. Hardening summary — what to mention, with mechanism

Each of these is a real, shipped property of the image — not aspirational:

| Property | Mechanism | Marketing-friendly framing |
|---|---|---|
| Signed pulls enforced on every host | cosign + `/etc/containers/policy.json` | "Hosts refuse to install an image we did not sign." |
| Management plane reachable only over Tailscale | firewalld zones (`tailscale` zone + restrictive `public` zone) | "Cockpit and remote management never touch the open internet — only your tailnet." |
| Updates fetch + stage, never auto-reboot | bootc timer + service drop-in | **Strong workstation differentiator.** "A 14-hour training run is never killed by an unattended security update — you reboot when you choose." Most cloud GPU products can't say this. |
| Rootless containers by default | rootful `podman.socket` disabled, rootless socket globally enabled | "Container workloads cannot escalate to root via the Docker socket." |
| SSH key-only, no root, no password | sshd drop-in | "SSH cannot be brute-forced; password and root login are off." |
| SELinux enforcing | `/etc/selinux/config` declared in image | "SELinux protects against process escape on every host." |
| Auto-update Flatpaks | `flatpak-system-update.timer` | "Firefox and other system flatpaks receive security updates daily without intervention." |
| Restricted default firewall zone | `/etc/firewalld/zones/public.xml` | "On any untrusted network, only SSH is reachable — everything else closed." |
| All CI dependencies SHA-pinned | every GitHub Action pinned by SHA + digest; Renovate keeps current | "Our build pipeline cannot be hijacked by a moving-tag action; a bot keeps pins current." |
| Annual signing-key rotation | `KEY-POLICY.md` + scheduled rotation routine | "We rotate our signing key annually, with a graceful transition window — never a hard cutover that breaks customer hosts." |
| CVE-scanned at every build | Grype in CI | "Every published image has its CVE report attached to the build run." |

## 4. The philosophy that shaped these choices

**Closed by default. The user opts into wider or riskier behavior with conscious, deliberate effort.**

Every choice above follows this pattern: close the untrusted surface, leave the user-controlled equivalent open, document the opt-in. This is the load-bearing principle and worth surfacing somewhere on the site — it explains to a sophisticated reader *why* the trade-offs look the way they do.

A few specific applications worth highlighting:

- **Updates** are *fetched and staged* automatically (closed default: no unattended reboots) but applied on the customer's schedule (open: their choice).
- **Cockpit** is closed to the LAN/internet (closed default) and open on the customer's tailnet (open: their own trust boundary).
- **Docker socket** is closed at the rootful (escalation-capable) level, open at the rootless (per-user, scoped) level.

This framing reads as thoughtful engineering, not security theater.

## 5. Verification recipes customers can run

These are powerful marketing assets. Putting them on the site means a customer's security team can validate the trust story in 30 seconds without taking the marketing's word for it. Suggested page-copy patterns:

```bash
# Anyone can verify the image we publish was signed by us:
cosign verify \
    --key https://raw.githubusercontent.com/rhuze-emryk/emryk-ml/main/cosign.pub \
    ghcr.io/rhuze-emryk/emryk-ml:latest

# Anyone can verify GitHub Actions built it from our source:
gh attestation verify \
    oci://ghcr.io/rhuze-emryk/emryk-ml:latest \
    --repo rhuze-emryk/emryk-ml

# Anyone can pull a complete bill of materials:
gh attestation verify \
    oci://ghcr.io/rhuze-emryk/emryk-ml:latest \
    --repo rhuze-emryk/emryk-ml \
    --predicate-type https://cyclonedx.org/bom
```

A "Verify" or "Trust" section on the site with these three blocks is a credible technical artifact a security team can paste into a terminal.

## 6. What NOT to claim

Be very careful here. Claims that overshoot are worse than no claim — they invite a security team to find the gap and lose trust.

Things the image **does not** protect against — do not claim these without per-customer caveats:

- **Per-workload isolation of ML jobs.** A malicious training script can read everything the user can read. The image does not sandbox ML jobs.
- **Full-disk encryption.** LUKS is the installer's decision, not the image's. If the customer needs encryption at rest, they configure it at install. Do not say "encrypted by default."
- **Customer-network isolation.** The image trusts whatever's on the tailnet. If a customer shares a tailnet with untrusted peers, that's their problem to solve.
- **GPU compute-side attacks.** Side channels between processes sharing a GPU are not addressed. Do not imply otherwise.
- **Anti-virus.** Intentionally not shipped. Linux endpoint AV is noise-to-signal-negative; we made a deliberate call. Don't claim AV.

`SECURITY.md` in the repo has the full "what we don't protect against" list. Treat it as marketing-binding — don't claim things the engineering document explicitly disclaims.

## 7. Tone guidance

- **Be specific, not generic.** "SLSA Level 3 build provenance" beats "secure builds." A real mechanism name beats a vague claim every time.
- **Lead with the workstation-friendly angles.** No-surprise-reboots and Tailscale-only-management are differentiators most competitors can't claim. Lead with those.
- **Show, don't tell.** Where possible, embed the verify commands rather than asserting trust. "Run this, see for yourself" is stronger than any marketing assertion.
- **Don't oversell.** Section 6 above is the line. Stay inside it.
- **Cross-link, don't duplicate.** When a customer wants the full threat model, link to `SECURITY.md` in the repo rather than restating it. The marketing site is the front door; the docs are the source of truth.

## 8. Working on the marketing site — practical pointers

When you write copy or page structure, useful structural ideas:

- A **"Trust" page** (or section under "How it works") with the three trust signals + verify recipes.
- A **"Operations" page** mentioning the fetch-only auto-updates as a customer-friendly differentiator.
- A **footer or about-page mention** of cosign / SLSA / SBOM with logos and one-sentence explanations linking out to those projects' homepages.
- A **link to `SECURITY.md`** on a security/legal page so security-savvy buyers can self-serve.

If you draft new claims that go beyond what's documented in `SECURITY.md`, run them past the maintainer (`rh@emryk.com`) first. The engineering posture documented in that file is the contract; marketing should not write checks it can't cash.
